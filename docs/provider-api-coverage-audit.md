All other Provider Integrations should still be treated as scoped Cognidesk support-channel adapters until they have the same proof. The remaining local module set mostly implements:

- provider manifests and Studio-visible capability metadata
- credential status/readiness helpers
- a small client surface for support-agent workflows
- conformance tests around manifests, credentials, signatures, and selected request construction

That is useful, but it is not equivalent to full coverage of each external partner's API.

The SDK now carries an explicit provider `coverage.scope` manifest field, exposes that scope in Studio Provider Integration rows, and fails provider conformance when a module claims `full-provider-api` without official URL evidence plus `metadata.fullProviderApiVerification` fields for provider, API version, verification date, operation catalog artifact, function catalog artifact, documented operation count, implemented operation count, generated function count, and zero unimplemented documented operations. Existing modules default to `support-workflow-subset` unless they explicitly declare a narrower or broader scope. Provider readiness also has explicit `scoped-verified` and `full-api-verified` states so a live check cannot quietly imply full external API coverage.

Generic request helpers, SDK-configured paths, or `request(path)` escape hatches do not count as full endpoint coverage. Full coverage requires one generated public callable per documented provider operation plus a checked artifact containing the official operation inventory.

For very large platform APIs such as Microsoft Graph, a package-specific generated slice can improve endpoint evidence without becoming `full-provider-api`. In that case the manifest must remain `provider-api-subset`, name the slice, and include generated operation/function counts without implying full platform coverage.

Provider conformance also now fails generic provider/action capabilities such as `send`, `handoff`, `transfer`, and Provider Object operations when they omit provider-object evidence. This keeps broad capability words tied to explicit external objects instead of becoming vague integration claims.

## Documentation-Backed Verification Rule

Provider tests that assert external request shape must cite or be traceable to the provider's original documentation. Mock-only green tests are not sufficient evidence for provider correctness.

Provider conformance is a contract guardrail, not proof of full partner API coverage. A passing conformance run means naming, manifest shape, capability vocabulary, coverage claims, credential status safety, provider-object evidence, fail-closed policy, and optional live-check behavior meet Cognidesk's SDK contract.

## Full Function Coverage Implementation Round

Date: 2026-06-17

The user clarified that `full-provider-api` must mean every exposable documented provider operation is reachable through its own SDK function. A generic path helper is useful escape-hatch plumbing, but it is not endpoint coverage.
