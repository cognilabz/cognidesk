# Publish external providers as one integration package

Superseded by ADR-0085. Official executable provider integrations now ship as independent SDK-backed provider/category packages with a shared integration kit and metadata-only catalog. Do not use `@cognidesk/integrations` or its category/provider subpaths for new provider runtime imports, and do not add an old-import compatibility bridge.

Official external provider integrations will be published from `@cognidesk/integrations` rather than as separate public provider packages. Provider-specific modules and provider-category modules such as Zendesk, Twilio Voice, OpenAI Realtime Voice, Gmail, ServiceNow, Genesys Cloud, forms, help center, community, and co-browsing live inside that package and are imported through category/provider subpath exports such as `@cognidesk/integrations/ticketing/zendesk`, `@cognidesk/integrations/contact-center/genesys-cloud`, and `@cognidesk/integrations/voice/openai`. The root `@cognidesk/integrations` export stays lightweight and limited to discovery metadata such as Provider Registry entries, manifests, categories, and shared types; provider clients and generated API surfaces are exported only from category/provider subpaths. Provider modules should stay dependency-light by default, using `fetch`, generated types, and local helpers where possible; rare heavy provider dependencies are optional peers that fail with clear setup errors only when that provider subpath is used. Tests and conformance evidence remain provider-specific even though distribution is unified, with additional registry/export-map tests for the Integration Package as a whole. SDK infrastructure and machinery packages such as `@cognidesk/core`, `@cognidesk/http`, `@cognidesk/storage`, `@cognidesk/model`, `@cognidesk/connections`, `@cognidesk/evals`, `@cognidesk/test-harness`, Studio packages, and the browser voice transport stay separate. Existing provider packages will be migrated directly into the Integration Package without compatibility shim packages or separate provider workspaces, so new and existing imports converge on the Integration Package surface. This supersedes ADR-0074 because the product surface should feel like one provider integration library with explicit module exports instead of many individually installed provider packages, while still preserving Provider Integration metadata, conformance evidence, and SDK-user-controlled enablement.

The migration scope includes community, contact-center, co-browsing, ecommerce, email, forms, help center, marketplace, messaging, reviews, SMS, social, ticketing, video, workplace, and provider/channel voice integrations such as OpenAI Realtime Voice, ElevenLabs, Twilio, Vonage, and SIP. It excludes Core, HTTP, storage, model, React, UI, OpenTelemetry, connections, evals, test-harness, Studio packages, and `voice-websocket`.

OpenAI Realtime currently uses static runtime and type imports from the official `openai` SDK, so the repository keeps `openai` available as a dev dependency for build, typecheck, and tests while publishing it only as an optional peer for consumers. Follow-up work should move the OpenAI adapter's runtime SDK access behind a small loader so a missing optional peer produces a provider-specific setup error instead of a generic module resolution failure.

Generated provider API surfaces are committed and published from the Integration Package. Maintainers regenerate them explicitly after checking upstream provider docs/specs; consumer installs must not run provider generation or fetch upstream specs. Large generated endpoint clients are split into smaller resource/tag modules behind stable generated compatibility files so the published subpath exports stay stable while endpoint functions remain reviewable.
