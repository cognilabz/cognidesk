# Split Studio web from Studio Operator Runtime

Cognidesk Studio will keep the web application boundary separate from the Studio Operator Runtime. The web app owns Studio UI, auth, manifests, dashboards, and browser-facing APIs, while the Studio Operator Runtime owns model execution, tool orchestration, source sandbox lifecycle, validation runs, and pull request preparation. The Operator Runtime is delivered as its own Docker image/container containing the Codex app server plus Cognidesk integration code; this keeps code-editing privileges and long-running agent work out of ordinary Studio routes while making the Codex-powered execution environment explicit. Studio authenticates to the Operator Runtime over a service-to-service channel and passes scoped per-session claims, Credential Grants, and allowed capabilities rather than letting the Operator Runtime own user login. The Studio web app owns the browser-facing WebSocket endpoint and proxies authenticated, normalized Studio Operator Event Stream messages to and from the internal Operator Runtime, which uses Codex app-server's WebSocket-capable notification transport. Local development should run the Studio Operator Runtime as a separate container so the production service boundary remains visible.
