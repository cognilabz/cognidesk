# Integrations Monolith Cleanup Map

Date: 2026-06-21

Issue: https://github.com/cognilabz/cognidesk/issues/27

This is the finishing checklist for deleting the legacy `@cognidesk/integrations` monolith and root provider-generation pipeline. It is intentionally a cleanup map, not a deletion patch. Runtime provider code under `packages/integrations/src/**` stays until the replacement provider package exists or `docs/provider-migration-matrix.md` explicitly says the provider is intentionally dropped.

## Final Deletion Gate

Do not remove legacy runtime until all of these are true for the provider being deleted:

- The owning tracker from #23-#25 or #29-#43 has landed a replacement package under `integrations/{category}/{provider}` or the matrix records `delete-not-migrated`.
- The replacement has manifest-only and runtime entry points, provider conformance coverage, catalog data, package-size budget metadata, and release smoke coverage.
- `pnpm providers:codemod:imports --check <changed paths>` reports no unhandled safe old imports for the provider, or leaves explicit review diagnostics for behavior changes.
- `@cognidesk/integration-catalog` prefers the split package manifest and the generated Provider Integration Catalog points at the split package, `/manifest`, and `/runtime` imports.
- CI verifies regenerated catalog data, legacy runtime-loader data, provider catalog docs, package smoke checks, and provider package size budgets without relying on the monolith.
- Application examples and docs use explicit registration and do not add an old-import bridge, compatibility package, runtime resolver, or `node_modules` provider scanner.
- Replacement packages use `@cognidesk/integration-{category}-{provider}` names. Provider fixtures or transitional packages named `@cognidesk/integration-*` are not allowed; `@cognidesk/integration-kit` and `@cognidesk/integration-catalog` remain provider-neutral infrastructure exceptions.

## Global Cleanup Checklist

Remove these surfaces only after the dependency gate is satisfied:

- `packages/integrations/package.json`, `tsconfig.json`, `src/index.ts`, `src/category-profiles.ts`, `src/provider-manifest.ts`, `src/internal/**`, and `test/**` once no provider runtime remains.
- Every provider export in `packages/integrations/package.json`, then the package root export itself. Current evidence has 66 public exports: root, `./category-profiles`, `./provider-manifest`, and 63 provider subpaths.
- `packages/integrations/src/provider-catalog/**` fallback loaders after catalog generation no longer needs legacy manifest/runtime fallbacks. The #28 workflow now fails if `packages/integrations/src/provider-catalog/runtime-loaders.generated.ts` is stale, so update generator expectations before removing that legacy generated file.
- Root generation commands in `package.json`: `providers:generate:list`, `providers:generate`, and `providers:split-generated` unless they are replaced by provider-package-local commands.
- Root generator scripts that write into `packages/integrations/src/**`, including `scripts/generate-*-full-api.mjs`, `scripts/generate-*-api.mjs`, `scripts/generate-*-inventory.mjs`, `scripts/generate-speech-provider-apis.mjs`, `scripts/generate-social-messaging-provider-coverage.mjs`, `scripts/generate-ticketing-provider-coverage.mjs`, `scripts/run-provider-generation.mjs`, `scripts/generate-provider-surfaces.mjs`, `scripts/harden-generated-client-types.mjs`, and `scripts/split-generated-client-modules.mjs`.
- Generated full-provider clones and generated chunk directories under legacy provider dirs, especially `full-api*.generated.ts`, `graph-api*.generated.ts`, `web-api*.generated.ts`, `http-api*.generated.ts`, `meetings-api*.generated.ts`, and `*.generated/client/*-chunk.generated.ts`.
- Generated-clone detection fixtures in `scripts/check-integration-package-architecture.test.mjs` after SDK-backed providers no longer need legacy clone examples. Keep fixtures for disallowed full-provider clone names and allowed metadata/support-slice generated files.
- Docs references that present `@cognidesk/integrations` as the runtime surface: `README.md`, `docs/provider-generation.md`, `docs/provider-api-coverage-audit.md`, `docs/releasing.md`, `website/api-reference/integrations.md`, `website/community/repository-map.md`, `website/concepts/transport-neutrality.md`, `website/guides/provider-packages.md`, `website/guides/provider-integrations-catalog.md`, `website/guides/provider-package-migration.md`, `website/guides/voice.md`, and generated API reference output.
- Generated catalog data and docs after split packages become authoritative: `packages/integration-catalog/src/catalog.generated.ts` and `website/guides/provider-integrations-catalog.md` must be regenerated, committed, and free of monolith runtime-package recommendations.
- Smoke expectations in `scripts/verify-package-smoke.mjs` that include `@cognidesk/integrations` by default. Keep split provider packages selected through `isProviderPackage`.
- Package-size budget expectations for the aggregate package. Keep `--fail-size-budget` for split provider packages and require provider-level `cognidesk.sizeBudget` metadata.
- CI references in `.github/workflows/integration-packages.yml` and `.github/workflows/publish.yml` that check the legacy package directly after split-package checks cover the migrated provider set. Preserve #28's generated catalog verification and `pnpm release:verify-packages -- --fail-size-budget` until replacement provider package checks fully cover the old surface.
